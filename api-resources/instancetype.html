Instancetype & Preference — VM 規格模板系統 ​

為什麼需要 Instancetype/Preference？ ​

在傳統的 KubeVirt 使用模式中，每個 VM 的 YAML 都必須完整指定 CPU 核心數、記憶體大小、磁碟 bus 類型、網路卡型號等所有設定。這導致大量重複配置，且難以統一管理規格標準。

Instancetype 與 Preference 系統解決了這個問題：

問題解決方案
每個 VM 重複指定相同規格Instancetype 定義標準規格
設備偏好設定重複Preference 定義偏好設定
規格不一致集中管理，強制標準化
難以批量更新規格修改 Instancetype 即可套用

概念區別

  • Instancetype：定義「硬性」資源需求 — CPU 和記憶體（用戶無法在 VM 中覆寫）
  • Preference：定義「軟性」偏好設定 — 設備類型、韌體等（若 VM 明確指定則覆寫）

資源類型對比 ​

Namespace 範圍 vs Cluster 範圍 ​

資源範圍用途
VirtualMachineInstancetypeNamespace給特定團隊/專案使用的規格
VirtualMachineClusterInstancetypeCluster全叢集共用的標準規格
VirtualMachinePreferenceNamespace給特定團隊/專案的偏好設定
VirtualMachineClusterPreferenceCluster全叢集共用的偏好設定

Instancetype Spec 詳細欄位 ​

yaml
apiVersion: instancetype.kubevirt.io/v1beta1
kind: VirtualMachineClusterInstancetype
metadata:
  name: cx1.medium
spec:
  cpu:
    # 分配給 VM 的 vCPU 數量（必填）
    guest: 2
    # 是否使用 CPU pinning（獨佔物理 CPU，避免超賣）
    dedicatedCPUPlacement: false
    # 是否隔離 emulator thread（減少 VM 負載對 QEMU 的影響）
    isolateEmulatorThread: false
    # 啟用 NUMA topology
    numa:
      guestMappingPassthrough: {}
    # Real-time VM（需要 RT kernel）
    realtime:
      mask: "0-1"

  memory:
    # 分配給 VM 的記憶體（必填）
    guest: "4Gi"
    # 記憶體超額分配上限（用於記憶體熱插拔）
    maxGuest: "8Gi"
    # 使用 HugePages（高效能場景）
    hugepages:
      pageSize: "1Gi"

  # IO Threads 策略（共享或自動）
  ioThreadsPolicy: auto

  # GPUS（直通 GPU 給 VM）
  gpus:
  - name: gpu1
    deviceName: vendor.com/GPU

  # Host Devices（PCI 直通）
  hostDevices:
  - name: fpga
    deviceName: vendor.com/FPGA

  # Launch Security（保密運算）
  launchSecurity:
    sev: {}

CPU 配置說明 ​

yaml
# 標準共享 CPU（最常見）
cpu:
  guest: 4

# CPU Pinning（高效能工作負載）
cpu:
  guest: 4
  dedicatedCPUPlacement: true
  isolateEmulatorThread: true  # 額外 1 個 CPU 給 QEMU

# Real-time CPU（需要 RT kernel，延遲敏感型應用）
cpu:
  guest: 2
  dedicatedCPUPlacement: true
  realtime:
    mask: "0-1"  # 指定哪些 vCPU 是 RT
  numaTopologySpread:
    cores: 1

記憶體配置說明 ​

yaml
# 固定記憶體
memory:
  guest: "8Gi"

# 記憶體熱插拔（配合 maxGuest）
memory:
  guest: "4Gi"
  maxGuest: "16Gi"

# HugePages（資料庫、低延遲應用）
memory:
  guest: "4Gi"
  hugepages:
    pageSize: "2Mi"  # 或 "1Gi"

Preference Spec 詳細欄位 ​

yaml
apiVersion: instancetype.kubevirt.io/v1beta1
kind: VirtualMachineClusterPreference
metadata:
  name: ubuntu
spec:
  # CPU 偏好
  cpu:
    preferredCPUTopology: preferSockets  # 或 preferCores, preferThreads, preferSpread, preferAny
    spreadOptions:
      across: CoresThreads
      ratio: 2

  # 設備偏好
  devices:
    preferredDiskBus: virtio          # 磁碟控制器類型
    preferredInterfaceModel: virtio   # 網路卡型號
    preferredInputBus: usb            # 輸入設備 bus
    preferredInputType: tablet        # 輸入設備類型（tablet 比 mouse 更適合 VNC）
    preferredDiskBlockSize:
      custom:
        logical: 512
        physical: 4096
    preferredRng: {}                  # 虛擬亂數產生器
    preferredDiskDedicatedIoThread: false
    preferredNetworkInterfaceMultiQueue: true  # 多佇列網路（效能提升）
    preferredTPM:                     # TPM 2.0（Windows 11 必須）
      persistent: true
    preferredAutoattachPodInterface: true

  # Feature 偏好（處理器特性）
  features:
    preferredAcpi: {}
    preferredApic: {}
    preferredHyperv:
      relaxed: {}
      vapic: {}
      spinlocks:
        enabled: true
        retries: 8191
      synic: {}
      synictimer:
        enabled: true
        direct: {}
      tlbflush: {}
      frequencies: {}
      reenlightenment: {}
      ipi: {}
      vendorid:
        enabled: true
        vendorid: "KVM HYPE"

  # 韌體偏好
  firmware:
    preferredUseBios: false
    preferredUseEfi: true
    preferredUseSecureBoot: false

  # Machine 偏好
  machine:
    preferredMachineType: q35  # 或 pc-i440fx（較舊）

  # 時鐘偏好
  clock:
    preferredClockOffset:
      utc: {}
    preferredTimer:
      hpet:
        present: false
      pit:
        tickPolicy: delay
      rtc:
        tickPolicy: catchup
      hyperv: {}

  # Subdomain 偏好（DNS 設定）
  preferredSubdomain: ""
  preferredTerminationGracePeriodSeconds: 180

CPU Topology 策略說明 ​

策略說明適用場景
preferSockets優先增加 socket 數量多數 Linux 工作負載
preferCores優先增加 core 數量Windows 授權最佳化
preferThreads優先增加 thread 數量（SMT/HT）HPC 場景
preferSpread均勻分散到所有拓撲層級一般用途
preferAny不限制拓撲彈性

Preference Requirements ​

Preference 可以定義最低規格需求，確保使用者建立的 VM 符合最低標準：

yaml
spec:
  requirements:
    cpu:
      guest: 2  # 至少需要 2 個 vCPU
    memory:
      guest: "4Gi"  # 至少需要 4 GiB 記憶體

需求驗證

若 VM 的 Instancetype 不符合 Preference 的 Requirements，VM 會被 Admission Webhook 拒絕建立。

在 VM 中引用 Instancetype/Preference ​

InstancetypeMatcher ​

yaml
apiVersion: kubevirt.io/v1
kind: VirtualMachine
metadata:
  name: my-vm
spec:
  instancetype:
    # 引用 Cluster 範圍的 Instancetype
    kind: VirtualMachineClusterInstancetype
    name: cx1.medium
    # revisionName 在建立 VM 後自動填入（版本追蹤）
    revisionName: cx1.medium-12345
    # inferFromVolume: 從 DataVolume/PVC 的 annotation 自動推斷
    # inferFromVolumeFailurePolicy: Ignore 或 RejectInferFromVolumeFailure

  preference:
    # 引用 Cluster 範圍的 Preference
    kind: VirtualMachineClusterPreference
    name: ubuntu
    revisionName: ubuntu-67890

引用 Namespace 範圍的資源 ​

yaml
spec:
  instancetype:
    # 省略 kind 時預設是 namespace 範圍
    kind: VirtualMachineInstancetype
    name: my-team-small
  preference:
    kind: VirtualMachinePreference
    name: my-team-centos

版本快照機制（ControllerRevision） ​

當 VM 引用 Instancetype/Preference 時，KubeVirt 會自動建立 ControllerRevision 快照：

bash
# 查看 VM 使用的 Instancetype 快照
kubectl get controllerrevision -l instancetype.kubevirt.io/name=cx1.medium

這確保即使 Instancetype 被更新，既有 VM 的行為不會改變（直到 VM 明確更新 revisionName）。

衝突解決機制 ​

衝突情況處理方式
VM spec 直接設定 CPU/Memory拒絕（Instancetype 是強制的）
VM spec 設定 diskBus（已有 Preference）VM 設定優先（Preference 是偏好）
VM spec 設定韌體（已有 Preference）VM 設定優先
VM spec 未設定任何偏好套用 Preference 設定

強制規則

Instancetype 定義的 CPU 和記憶體是強制性的，VM spec 中不能同時設定這些值，否則建立會失敗。

定義 Instancetype ​

yaml

kind: VirtualMachineClusterInstancetype

  name: standard.small
  labels:
    instancetype.kubevirt.io/vendor: "platform-team"
    instancetype.kubevirt.io/version: "v1"

  cpu:
    guest: 2
  memory:
    guest: "4Gi"
---
apiVersion: instancetype.kubevirt.io/v1beta1
kind: VirtualMachineClusterInstancetype
metadata:
  name: standard.medium
spec:
  cpu:
    guest: 4
  memory:
    guest: "8Gi"
---
apiVersion: instancetype.kubevirt.io/v1beta1
kind: VirtualMachineClusterInstancetype

  name: high-performance.large

  cpu:
    guest: 8
    dedicatedCPUPlacement: true
    isolateEmulatorThread: true

    guest: "32Gi"
    hugepages:
      pageSize: "1Gi"

使用 Instancetype 和 Preference 建立 VM ​

yaml
apiVersion: kubevirt.io/v1
kind: VirtualMachine
metadata:
  name: ubuntu-server
  namespace: default
spec:
  running: true

  # 引用 Instancetype（強制規格）
  instancetype:
    kind: VirtualMachineClusterInstancetype
    name: standard.medium  # 4 vCPU, 8Gi 記憶體

  # 引用 Preference（偏好設定）
  preference:
    kind: VirtualMachineClusterPreference
    name: ubuntu-22.04

  template:
    metadata:
      labels:
        kubevirt.io/vm: ubuntu-server
    spec:
      # 注意：不需要在 domain 中設定 CPU/Memory（由 Instancetype 決定）
      domain:
        devices:
          disks:
          - name: rootdisk
            disk:
              # bus 由 Preference 決定（virtio）
          - name: cloudinitdisk
            disk: {}
          interfaces:
          - name: default
            masquerade: {}
      networks:
      - name: default
        pod: {}
      volumes:
      - name: rootdisk
        containerDisk:
          image: quay.io/containerdisks/ubuntu:22.04
      - name: cloudinitdisk
        cloudInitNoCloud:
          userData: |
            #cloud-config
            password: password
            chpasswd: { expire: False }

virtctl 管理 Instancetype ​

bash
# 建立 Instancetype（互動式）
virtctl create instancetype --name standard.small --cpu 2 --memory 4Gi

# 建立 ClusterInstancetype
virtctl create instancetype \
  --name high-perf.xlarge \
  --cpu 16 \
  --memory 64Gi \
  --cpu-pinning \
  --cluster-scope

# 建立 Preference
virtctl create preference \
  --name ubuntu \
  --preferred-cpu-topology preferSockets \
  --preferred-disk-bus virtio \
  --preferred-interface-model virtio \
  --cluster-scope

常用操作指令 ​

bash
# 列出所有 ClusterInstancetype
kubectl get virtualmachineclusterinstancetype

# 列出所有 ClusterPreference
kubectl get virtualmachineclusterpreference

# 查看 VM 使用的 Instancetype 詳情
kubectl get vm my-vm -o jsonpath='{.spec.instancetype}'

# 查看版本快照
kubectl get controllerrevision \
  -l instancetype.kubevirt.io/name=standard.medium

# 更新 VM 使用的 Instancetype 版本
kubectl patch vm my-vm --type merge \
  -p '{"spec":{"instancetype":{"revisionName":""}}}'
# 刪除 revisionName 後，下次 VM 重啟會使用最新版本

# 套用新的 Instancetype 配置
kubectl apply -f updated-instancetype.yaml

最佳實踐

建議平台團隊統一管理 ClusterInstancetype 和 ClusterPreference，讓應用程式團隊只需指定 instancetype: standard.small 即可，不需要了解底層硬體細節。
